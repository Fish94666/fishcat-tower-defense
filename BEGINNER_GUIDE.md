# 初學者導讀：Fishcat Tower Defense

這份文件是搭配這個專案一起看的。建議你一邊開著 [index.html](C:\Users\fish\codes\idk\index.html)、[style.css](C:\Users\fish\codes\idk\style.css)、[main.js](C:\Users\fish\codes\idk\main.js)，一邊對照這份說明。

## 1. 先看整體架構

- [index.html](C:\Users\fish\codes\idk\index.html) 是畫面骨架。它放了標題、按鈕、狀態區，還有真正拿來畫遊戲的 `<canvas>`.
- [style.css](C:\Users\fish\codes\idk\style.css) 是外觀。它決定版面、顏色、按鈕樣式、畫布大小。
- [main.js](C:\Users\fish\codes\idk\main.js) 是遊戲大腦。敵人怎麼走、塔怎麼打、點滑鼠怎麼放塔、每一幀要重畫什麼，都在這裡。

你可以先記一個簡單分工：

- `HTML`：放元件
- `CSS`：讓元件好看
- `JavaScript`：讓元件動起來

## 2. `main.js` 該怎麼讀

最適合初學者的順序不是從第一行看到最後一行，而是照「資料 -> 狀態 -> 規則 -> 畫面 -> 主迴圈」去看。

### A. 初始化與常數

重點位置：`canvas`、`ctx`、`statsEl`、`messageEl`、`startBtn`，以及 `TILE_SIZE`、`GRID_COLS`、`STARTING_GOLD` 這些常數。

這段在做什麼：
- 先抓到 HTML 裡的元素，讓 JavaScript 能控制它們。
- 再把地圖大小、金錢、生命值、塔價錢這些固定規則集中寫在前面。

為什麼要這樣寫：
- 常數集中在一起，之後調整平衡時比較不會到處找。
- 先把 DOM 元素存起來，後面不用一再重抓。

初學者最容易卡住的點：
- `canvas` 是畫布元素，`ctx` 才是拿來畫圖的工具。
- `const` 不是完全不能變，而是「這個變數名稱不能重新指向別的東西」。

最安全的改法：
- 想調難度時，先改常數，例如 `STARTING_GOLD` 或 `TOWER_COST`。
- 不要一開始就改 `canvas` 尺寸，因為它會連動格子座標與滑鼠換算。

### B. 遊戲資料

重點位置：`mapLayout`、`pathCells`、`pathPoints`、`waves`、sprite 設定。

這段在做什麼：
- `mapLayout` 定義每個格子是草地、路徑還是石頭。
- `pathCells` 定義敵人要走過哪些格子。
- `pathPoints` 把格子座標換成畫布上的像素座標，方便移動敵人。
- `waves` 定義每一波要出幾隻敵人、什麼種類、間隔多久出一隻。
- sprite 設定告訴程式圖片在哪裡，以及圖片最大要畫多大。

為什麼要這樣寫：
- 地圖資料和遊戲規則分開存，後面比較好維護。
- 先把「格子座標」換成「像素座標」，敵人移動時會單純很多。

初學者最容易卡住的點：
- 格子座標和像素座標是兩套系統，不要混在一起。
- `map()` 是把陣列中的每個資料轉成新資料，不會直接修改原陣列。

最安全的改法：
- 想改地圖時，先同步檢查 `mapLayout` 和 `pathCells` 有沒有互相對上。
- 想加新波次時，先照現有 `waves` 格式新增一筆，不要先改生成邏輯。

### C. 狀態管理

重點位置：`resetState()`、`state`、`startGame()`。

這段在做什麼：
- `resetState()` 建立一份全新的初始遊戲狀態。
- `state` 是現在整場遊戲的資料中心，像金幣、生命、塔、敵人都放這裡。
- `startGame()` 會把狀態重設，然後切成 `playing`.

為什麼要這樣寫：
- 把初始值集中在 `resetState()`，重新開始時只要呼叫一個函式就好。
- 用單一 `state` 管理全部資訊，比把資料散在很多全域變數更清楚。

初學者最容易卡住的點：
- `state` 會一直變，這種資料叫做「狀態」。
- `mode` 很重要，它控制現在是開始前、進行中、勝利還是失敗。

最安全的改法：
- 想新增資源或新狀態時，先加進 `resetState()`，再找所有會更新 HUD 的地方。
- 不要在很多地方偷偷塞新的全域變數，先考慮能不能放進 `state`.

### D. 核心邏輯

重點位置：`updateWave()`、`updateEnemies()`、`updateTowers()`、`updateEffects()`、`update()`.

這段在做什麼：
- `updateWave()` 決定什麼時候生怪、什麼時候進下一波、什麼時候勝利。
- `updateEnemies()` 讓敵人沿著路徑往下個節點移動，走到底就扣生命。
- `updateTowers()` 找射程內最合適的敵人，轉向、攻擊、加金幣。
- `updateEffects()` 讓短暫的攻擊光線慢慢消失。
- `update()` 把上面幾個更新步驟串起來。

為什麼要這樣寫：
- 每個函式只負責一種系統，讀起來比較不亂。
- 遊戲更新邏輯和畫面繪製分開，除錯時比較好追。

初學者最容易卡住的點：
- `dt` 是「這一幀經過了多少秒」，很多移動都靠它讓速度穩定。
- 敵人不是瞬間跳格，而是每一幀朝下一個 `pathPoint` 前進一小步。
- 塔攻擊不是每幀都打，`cooldownLeft` 會限制射擊頻率。

最安全的改法：
- 想調平衡時，先改 `damage`、`cooldown`、`speed` 這種數值。
- 不要同時改 `updateWave()` 和 `updateEnemies()`，一次只動一個系統比較容易除錯。

### E. 使用者互動

重點位置：`getPointerCell()`、`placeTower()`、事件監聽器。

這段在做什麼：
- `getPointerCell()` 把滑鼠點到的畫面位置換成地圖格子。
- `placeTower()` 檢查能不能蓋塔，再扣錢並建立新塔。
- `addEventListener()` 把滑鼠移動、離開、點擊、按按鈕這些互動接到對應函式。

為什麼要這樣寫：
- 使用者看到的是像素座標，但遊戲規則多半用格子座標判斷。
- 檢查是否合法再放塔，可以避免不小心蓋在路上或石頭上。

初學者最容易卡住的點：
- `canvas` 在畫面上可能被 CSS 縮放，所以要用 `scaleX`、`scaleY` 換算。
- 事件監聽器本身不做太多事，而是把工作交給函式處理。

最安全的改法：
- 想加新互動時，盡量照現在的模式做：事件只負責收資料，邏輯交給函式。
- 想改放塔條件時，優先改 `canBuildAt()` 和 `placeTower()`.

### F. 畫面渲染

重點位置：`drawMap()`、`drawTowers()`、`drawEnemies()`、`drawOverlay()`、`draw()`.

這段在做什麼：
- `drawMap()` 畫地圖和滑鼠 hover 高亮。
- `drawTowers()` 畫塔，有圖片就畫圖片，沒有就畫備用圖形。
- `drawEnemies()` 畫敵人和血條。
- `drawEffects()` 畫短暫的攻擊線。
- `drawOverlay()` 在未開始、勝利、失敗時加一層提示。
- `draw()` 把所有繪圖步驟依順序組起來。

為什麼要這樣寫：
- 畫面是有先後順序的，先畫地圖，再畫角色，最後畫 UI，才不會被蓋掉。
- 有 fallback 畫法代表就算圖片沒載入，遊戲仍然能玩。

初學者最容易卡住的點：
- `ctx.save()` / `ctx.restore()` 是在保護畫圖狀態，避免旋轉或透明度影響後面的東西。
- `clearRect()` 不是清除某個角色，而是先把整張畫布清空再整張重畫。

最安全的改法：
- 想改外觀時，先從顏色、大小、血條長度開始。
- 想改繪製順序時要小心，因為順序一變就可能把物件蓋住。

### G. 主迴圈

重點位置：`animationLoop()`、`requestAnimationFrame()`.

這段在做什麼：
- `animationLoop()` 每一幀都計算經過多少時間，更新遊戲，再重畫畫面。
- `requestAnimationFrame()` 會請瀏覽器在下一幀再呼叫一次，形成循環。

為什麼要這樣寫：
- 這是網頁遊戲很常見的基本骨架。
- 把 `dt` 限制在 `0.05` 以內，可以避免某一幀卡太久時角色突然暴衝。

初學者最容易卡住的點：
- 主迴圈不是 `for` 迴圈，它是瀏覽器一幀一幀幫你叫回來。
- 遊戲看起來在「持續執行」，其實是很多次很快的更新加重畫。

最安全的改法：
- 想加新系統時，先決定它應該放在 `update()` 還是 `draw()`.
- 不要在主迴圈內做太重的工作，不然畫面會卡。

## 3. 這四個公開入口要特別看

### `startGame()`

這是「重新開始遊戲」的入口。按下按鈕時會呼叫它。你可以把它理解成整場遊戲的 reset 按鈕。

### `placeTower(gridX, gridY)`

這是「在某格放塔」的入口。滑鼠點擊後，會把點到的格子座標丟進來。

### `render_game_to_text()`

這是為了測試和除錯準備的。它會把目前的遊戲狀態整理成一段 JSON 字串，讓工具或開發者不用只靠畫面，也能知道遊戲現在發生什麼事。

### `advanceTime(ms)`

這是讓程式「手動前進時間」的工具。測試時很好用，因為可以不等真實時間流逝，就直接推進模擬。

## 4. 為什麼 `window.__towerDefense` 很重要

在瀏覽器裡，`window` 可以想成全域空間。把 `startGame`、`placeTower`、`getState` 掛到 `window.__towerDefense` 上，有兩個主要好處：

- 除錯方便：你可以在瀏覽器開發者工具 console 裡直接呼叫它們。
- 測試方便：自動化工具可以直接讀取遊戲狀態，或觸發某些操作。

你可以把它理解成「專門留給開發者用的小控制台 API」。

## 6. 你可以自己做的安全練習

這些練習都很適合初學者，而且不容易把整個專案改壞：

1. 把初始金錢改成 `150`，觀察一開始能不能多放幾座塔。
2. 把塔的 `damage` 從 `11` 改大，看看敵人是不是更快被消滅。
3. 把某一波的 `count` 改小，確認難度是否下降。
4. 把草地或路徑顏色改掉，練習分辨「邏輯」和「外觀」的差別。
5. 在 `message` 加入自己的提示文字，練習找狀態更新點。

## 7. 用實際操作驗證你有沒有看懂

你可以拿下面這份清單來檢查自己是不是已經抓到主要邏輯：

- 按下 Start 後，`state.mode` 會變成 `playing`
- 點草地可以放塔，而且 `gold` 會減少
- 敵人會照 `pathCells` 的順序走
- 塔進入射程後會攻擊敵人，畫面上會出現短暫攻擊線
- 敵人走到底會扣 `lives`
- 全部波次結束後，遊戲會進入 `won`
